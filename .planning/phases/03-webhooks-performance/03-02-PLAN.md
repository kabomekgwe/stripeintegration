## Admin Service N+1 Query Issue

In `getUsersList()` method (lines 299-336), the current implementation:
1. Queries users with pagination
2. Includes `_count` for payments (efficient - uses Prisma's count feature)
3. Includes `payments` relation with `take: 1` to get last payment
4. Maps over users and calculates totalSpent by reducing payments array

The problem: For each user, Prisma loads the payments relation separately, causing N+1 queries.

## Current Schema Indexes

Missing indexes needed:

- Subscription userId and status lookups
- WebhookEvent indexes for dashboard queries
- Refund paymentId for lookups

## Caching Strategy

Frequently accessed data that should be cached:
1. User profiles (by userId) - TTL: 5 minutes
2. Payment records (by paymentId) - TTL: 2 minutes
3. Subscription data (by subscriptionId) - TTL: 1 minute
4. Recent webhook events (aggregated) - TTL: 30 seconds

Cache key patterns:
- `user:{userId}` - User profile data
- `payment:{paymentId}` - Payment record
- `subscription:{subscriptionId}` - Subscription data
- `webhook:stats` - Webhook statistics

    Fix the N+1 query in `getUsersList()` method by replacing the per-user payment query with a more efficient approach:

    Current problematic code (lines 299-336):
    ```typescript
    include: {
      _count: { select: { payments: true } },
      payments: {
        where: { status: PaymentStatus.SUCCEEDED },
        select: { amount: true, createdAt: true },
        orderBy: { createdAt: 'desc' },
        take: 1,
      },
    },
    ```

    Solution options:
    1. Use Prisma's `aggregateRaw` or raw query with JOIN for user list with aggregated payment data
    2. Separate query: First get users, then batch fetch payment aggregates
    3. Use Prisma's `groupBy` on payments and join in application code

    Recommended approach (option 2 - batch fetch):
    - Query users without payment includes
    - After getting user IDs, query PaymentRecord with groupBy userId to get totals
    - Query PaymentRecord for last payment dates separately
    - Merge results in the map function

    This reduces from N+1 queries to 3 queries total (users, payment totals, last payments).

    Also review `getUserDetails()` method (lines 338-373) for similar issues with includes.
