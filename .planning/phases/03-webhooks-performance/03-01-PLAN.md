## WebhooksService Overview

The WebhooksService handles Stripe webhook events with the following key features:
- Signature verification using Stripe webhook secret
- Event persistence to database
- Redis-based distributed locking to prevent duplicate processing
- Type-guard based event routing to specific handlers
- Support for payment_intent, setup_intent, subscription, dispute, and account events
- Dashboard methods for stats, event listing, and retry functionality

## Key Methods to Test

1. `processWebhook(payload, signature)` - Main entry point
2. `handleEvent(event)` - Event routing
3. `handlePaymentIntentEvent` / `handlePaymentIntentSucceeded` / `handlePaymentIntentFailed`
4. `handleSetupIntentEvent` / `handleSetupIntentSucceeded` / `handleSetupIntentFailed`
5. `handleSubscriptionEvent` / `handleSubscriptionCreated` / `handleSubscriptionUpdated` / `handleSubscriptionDeleted`
6. `handleDisputeEvent` / `handleDisputeCreated` / `handleDisputeUpdated`
7. `handleAccountEvent` / `handleAccountUpdated`
8. `getWebhookStats()` - Dashboard stats
9. `getWebhookEvents(params)` - Paginated event listing
10. `getWebhookEvent(id)` - Single event retrieval
11. `retryWebhookEvent(id)` - Retry failed events
12. `getRecentErrors(limit)` - Error listing

## Dependencies to Mock

- PrismaService (webhookEvent.create, update, findMany, count, groupBy)
- StripeService (constructWebhookEvent)
- RedisService (acquireWebhookLock, releaseWebhookLock)
- ConfigService (STRIPE_WEBHOOK_SECRET)
- SubscriptionService (handleStripeSubscriptionUpdated, handleStripeSubscriptionDeleted)
- DisputeService (handleDisputeCreated, handleDisputeUpdated)
- ConnectService (handleAccountUpdated)
