<decisions_made>

- **Error Handling Architecture**: Decided to create separate `stripe.errors.ts` module rather than adding to existing service. This allows:
  - Clean separation of concerns
  - Easy import from any service that calls Stripe
  - Standalone unit testing
  - Reusability across payments, webhooks, subscriptions services

- **HTTP Status Mapping**: Decided to use NestJS HttpException with proper status codes rather than custom error middleware:
  - `BAD_REQUEST` (400) for card/invalid request errors
  - `TOO_MANY_REQUESTS` (429) for rate limits
  - `INTERNAL_SERVER_ERROR` (500) for auth/api errors
  - `SERVICE_UNAVAILABLE` (503) for connection errors

- **User-Friendly Messages**: Decided to maintain both technical `message` and user-facing `userMessage`:
  - Technical messages logged for debugging
  - User messages returned to API consumers for display
  - Decline code mapping for common card errors

</decisions_made>
